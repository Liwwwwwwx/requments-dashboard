# Product

## Register

product

## Users

内部工程团队与项目管理者（研发负责人、PM、QA、各角色 owner）。使用场景是在工作日的协作流中快速掌握「事情进行到哪了」：哪些需求在进行、哪些被阻塞、谁负责什么、本周要交付什么。多为桌面端长时间停留，常与代码编辑器、终端并排打开。

## Product Purpose

TraceBoard 是事件溯源驱动的多项目需求管理平台。它把需求与任务的状态变更记录为事件流，再渲染成看板视图，让团队对需求生命周期（待开始 / 进行中 / 暂停 / 完成）一目了然。平台会扩展到更多业务模块（数据仪表盘 / 分析、管理后台 / 设置、更多项目视图），因此应用框架需要从一开始就支持多模块导航，而不是只服务单一看板。成功的标准是：成员打开就能在几秒内定位状态、找到自己负责的事项，工具本身不抢戏。

## Brand Personality

精确、克制、专业（precise / restrained / professional）。语气像一位资深工程师：直接、可信、不浮夸。界面应当传递「掌控感」与「清晰」，而非热闹或营销感。三个词：clear, calm, dependable。

## Anti-references

- 不要营销落地页式的视觉：大号 hero 标题、渐变文字、装饰性插画、营销 buzzword。
- 不要花哨仪表盘的「大数字 + 渐变卡片」套路。
- 不要 Trello 那种高饱和彩色卡片堆砌、玩具感。
- 不要把工具做成「好看但难用」：动效不抢戏、不发明非标准控件。
- 不要 Inter 直接套 Linear 的克隆感——保留自有的字体与蓝色身份。

## Design Principles

- **工具隐入任务**：信息密度优先，每个像素服务于「读懂状态」，装饰让位于可读性。
- **一致优于惊喜**：同一种按钮、表单、图标、间距语言贯穿所有模块；delight 留给瞬间，不铺满页面。
- **为扩展而建**：框架（顶栏 + 导航侧栏）是可复用的外壳，新业务模块是配置 + 路由的增量，不是重写。
- **状态即色彩**：颜色承载语义（优先级、状态、阻塞），不做无意义的色彩装饰。
- **诚实的预留**：未上线的模块明确标记，而非伪装成可用。

## Accessibility & Inclusion

- 目标 WCAG 2.1 AA：正文对比度 ≥4.5:1，大字 / 图标 ≥3:1，占位符同样达 4.5:1（亮、暗两套都需核对）。
- 完整键盘可达：看板卡片、导航、切换器均可 Tab / Enter 操作，可见焦点环。
- 颜色不作为唯一信息载体：状态 / 优先级除颜色外辅以文字或形状。
- 亮 / 暗双主题 + 跟随系统；尊重 `prefers-reduced-motion`，所有动效提供降级。
