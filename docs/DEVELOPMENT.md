## 开发流程

```text
需求讨论 → 编写功能文档 → Review 文档 → 实现 → 自测 → 合并
```

1. **讨论**：在 Issue 或对话中明确要做什么
2. **编写文档**：根据提示生成 `docs/features/<编号>-<名称>.md`
3. **明确验收标准**：文档中必须写清楚做完哪些才算完成
4. **实现**：按文档中分解的步骤逐步完成
5. **自测**：开发过程中保持前后端构建通过

## 文档约定

### 编号规则

`<分类缩写>-<功能名>`

| 缩写 | 分类 |
|------|------|
| FE | 前端体验 |
| BE | 后端能力 |
| UX | 交互/布局 |
| DX | 开发体验 |
| ARCH | 架构变更 |

示例：`UX-right-panel.md`、`BE-state-snapshot.md`、`DX-event-log.md`

### 状态标记

| 状态 | 含义 |
|------|------|
| proposed | 刚提出，未开始 |
| in-progress | 正在开发 |
| done | 已完成 |
| on-hold | 暂停（依赖未就绪等） |

### 文档结构

每个功能文档固定包含：

1. **目标与动机** — 一句话说清楚做什么、为什么
2. **用户场景** — 谁在什么情况下用
3. **前端范围** — 页面/路由/组件/交互状态
4. **后端范围** — API/事件/数据模型/状态变更
5. **API 契约**（如有）— 请求/响应结构
6. **实现步骤** — 按顺序拆分，标注前后端依赖
7. **验收标准** — 完成清单
